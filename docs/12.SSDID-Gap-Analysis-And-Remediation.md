# SSDID Gap Analysis & Remediation Plan

## 1. Overview

This document identifies critical architectural gaps in the SSDID system and proposes proven remediation approaches for each. Gaps are prioritized by severity and grouped into implementation phases.

## 2. Gap Map

| # | Gap | Severity | Impact if Unsolved |
|---|---|---|---|
| 1 | Key recovery | **Critical** | Identity permanently lost on device/password loss |
| 2 | Key rotation | **Critical** | Compromised key = compromised identity forever |
| 3 | Multi-device identity | **High** | Users locked to single device |
| 4 | Credential revocation | **High** | Issued credentials cannot be invalidated |
| 5 | Backup & export | **Critical** (mobile) | No disaster recovery for key material; on mobile, device loss = permanent identity loss (same as Gap 1) |
| 6 | Offline verification | **Medium** | Verification fails without network connectivity |
| 7 | Registry high availability | **Medium** | Single-node Mnesia = single point of failure |
| 8 | Rate limiting | **Medium** | DID enumeration and resource abuse |
| 9 | Activity history persistence | **Medium** | No audit trail for identity/credential operations |
| 10 | Credential status model | **Medium** | `VerifiableCredential` model has no `credentialStatus` field — prerequisite for Gap 4 |

## 3. Current State Summary

### What Exists — Backend (Elixir Registry)

- **Key storage**: Password-protected keys with Argon2id KDF, HMAC integrity, disk persistence
- **Multi-key DID Documents**: Up to 10 verification methods per DID (configurable)
- **Challenge-response authentication**: Single-use, time-bound, domain-verified
- **Mnesia storage**: ACID transactions, disc_copies for DID Documents
- **19 cryptographic algorithms**: Classical (Ed25519, ECDSA P-256/P-384) + PQC (ML-DSA, SLH-DSA, KAZ-Sign)
- **Proof verification**: Freshness validation, purpose matching, constant-time comparison

### What Exists — Mobile App (Android)

- **Key storage**: Hardware-backed via Android Keystore (TEE/StrongBox). KAZ-Sign private keys encrypted with AES-256-GCM wrapping key from hardware keystore. No Argon2id (too resource-heavy for mobile).
- **6 algorithm variants**: ED25519, ECDSA_P256, ECDSA_P384, KAZ_SIGN_128, KAZ_SIGN_192, KAZ_SIGN_256
- **Biometric authentication**: BIOMETRIC_STRONG gated key operations
- **Single-key DID Documents**: Each identity creates one key, one DID Document
- **Identity/credential storage**: DataStore Preferences (metadata) + filesDir (encrypted keys)
- **URL validation**: HTTPS-only, SSRF-safe server URL validation
- **Online only**: No offline verification capability (design decision)

### What Is Missing

- No key recovery mechanism (documented in Vault Spec Section 6, not implemented)
- No key rotation ceremony or pre-commitment
- No multi-device enrollment protocol
- No credential revocation registry (and `VerifiableCredential` model lacks `credentialStatus` field)
- No encrypted backup/export format (hardware-bound keys cannot be exported directly — see Gap 5)
- No offline verification bundle (mobile design doc states "Online only")
- No rate limiting (and mobile app has no HTTP 429 handling)
- No activity history persistence layer
- No `nextKeyHash` field in `DidDocument` model (prerequisite for Gap 2)
- Challenge store uses ram_copies (lost on restart)

## 4. Gap 1: Key Recovery (Critical)

### Problem

Loss of device or password results in permanent identity loss. This is the primary reason enterprises reject self-sovereign identity systems.

### Approach: Three-Tier Recovery

#### Tier 1: Recovery Key (Default)

At identity creation, generate a separate recovery keypair. The recovery public key is registered in the DID Document. The recovery private key is stored offline by the user (printed QR code, USB drive, safe deposit box).

```
DID Document:
  verificationMethod:
    - #device-key   (daily use, stored in device vault)
    - #recovery-key (emergency only, stored offline)
  authentication: [#device-key]
  capabilityInvocation: [#device-key, #recovery-key]
```

Recovery flow:
1. User retrieves offline recovery key
2. Signs a DID Document update using recovery key
3. Removes old device key, adds new device key
4. Recovery key itself remains unchanged

Reference implementations: Sovrin, ION (Bitcoin-anchored DID)

#### Tier 2: Social Recovery (Shamir's Secret Sharing)

Split the recovery secret into N shares using Shamir's Secret Sharing over GF(256). Distribute shares to trusted contacts. Require K-of-N shares to reconstruct (e.g., 3-of-5).

```
Setup:
  1. Generate master recovery secret
  2. Split into 5 shares via Shamir's Secret Sharing
  3. Encrypt each share to the guardian's public key
  4. Distribute encrypted shares to 5 trusted contacts/devices/institutions

Recovery:
  1. Collect 3 encrypted shares from guardians
  2. Each guardian decrypts their share
  3. Reconstruct master recovery secret from 3 shares
  4. Use recovery secret to sign key rotation on DID Document
```

Design constraint: Shares must be encrypted to each guardian's public key so guardians cannot collude without the recovery ceremony. Share encryption uses ECIES (for classical guardian keys) or KEM+AES-256-GCM (for PQC guardian keys), determined by the guardian's key algorithm.

Reference implementations: Argent wallet (Ethereum), Ledger Recover, W3C DID Recovery spec draft

#### Tier 3: Institutional Recovery (Enterprise)

An organization (employer, government, bank) acts as a recovery authority. They hold a recovery key registered in the DID Document with a dedicated purpose.

```
DID Document:
  verificationMethod:
    - #user-key
    - #org-recovery-key (controller: did:ssdid:org123)
  capabilityInvocation: [#user-key, #org-recovery-key]
```

The organization can rotate the user's key on request, subject to the organization's own identity verification procedures. This re-introduces trust in a third party but is opt-in, not architectural.

**Exit protocol**: If the user leaves the organization or revokes the institutional recovery relationship, the user signs a DID Document update removing `#org-recovery-key` from `verificationMethod` and `capabilityInvocation`. The organization cannot block removal because the user's own key retains `capabilityInvocation` rights. If the user's key is lost and only the org key remains, the org performs a final rotation to a user-controlled key before removal.

Reference implementations: Microsoft ION (enterprise mode), Hyperledger Indy (guardianship)

### Recommended Strategy

All three tiers should be supported. Selection is per-identity:

| Use Case | Recommended Tier |
|---|---|
| Individual user (default) | Tier 1: Recovery key |
| Privacy-conscious user | Tier 2: Social recovery |
| Enterprise employee | Tier 3: Institutional recovery |
| High-security | Tier 1 + Tier 2 combined |

## 5. Gap 2: Key Rotation (Critical)

### Problem

The architecture supports multiple keys in a DID Document, but there is no ceremony for safely rotating from a compromised key to a new key. An attacker who obtains the current key can race the legitimate owner to rotate first.

### Approach: Pre-Rotation (KERI Protocol)

When a key is created, the owner also commits to the next key by publishing a hash of the next key's public key. An attacker who steals the current key cannot rotate to their own key because the next key was pre-committed.

```
Step 1: Identity creation
  - Generate key_0 (active) and key_1 (pre-rotated, stored securely)
  - DID Document includes: key_0 + SHA3-256(key_1.public)

Step 2: Normal rotation
  - Reveal key_1, generate key_2
  - Sign rotation with key_0 (current)
  - New DID Document: key_1 + SHA3-256(key_2.public)
  - key_0 is retired

Step 3: Emergency rotation (key_0 compromised)
  - Attacker has key_0, but cannot forge key_1 (only hash was published)
  - Owner uses key_1 to rotate
  - Attacker's rotation attempt rejected: proposed new key doesn't match pre-committed hash
```

### DID Document Extension

```json
{
  "id": "did:ssdid:abc123",
  "verificationMethod": [
    {"id": "#key-1", "publicKeyMultibase": "u...", "type": "Ed25519VerificationKey2020"}
  ],
  "capabilityInvocation": ["#key-1"],
  "nextKeyHash": "uSHA3_256_of_next_public_key_base64url"
}
```

### Policy Additions

| Parameter | Default | Description |
|---|---|---|
| `rotation_grace_period_seconds` | 300 | Both old and new key valid during transition (5 minutes — shorter minimizes attack window) |
| `max_rotation_frequency` | 1 per hour, per DID | Prevents rapid-fire rotation attacks. Enforced per-DID globally, not per-key. |
| `require_pre_rotation` | true | Reject rotations without matching pre-commitment |

### Mobile Model Changes Required

- `DidDocument.kt`: Add `nextKeyHash: String? = null` field
- `Identity.kt`: Add `preRotatedKeyId: String? = null` for tracking pre-committed next key
- `VaultStorage`: New methods for storing pre-rotated key material (second encrypted key file per identity)
- `Vault`: New methods `prepareRotation()` and `executeRotation()`
- `VerifierImpl`: Ensure `Json { ignoreUnknownKeys = true }` to handle DID Documents with `nextKeyHash`

Reference: KERI (Key Event Receipt Infrastructure), gaining adoption in EU eIDAS 2.0

## 6. Gap 3: Multi-Device Identity (High)

### Problem

A user with phone, tablet, and laptop needs separate DIDs for each device. There is no way to use the same identity across devices.

### Approach: Device-Bound Keys Under One DID

Each device gets its own keypair, all registered under the same DID Document. The DID Document already supports multiple verification methods — this gap requires an enrollment protocol, not a schema change.

```json
{
  "id": "did:ssdid:abc123",
  "verificationMethod": [
    {"id": "#phone-key",  "publicKeyMultibase": "u..."},
    {"id": "#tablet-key", "publicKeyMultibase": "u..."},
    {"id": "#laptop-key", "publicKeyMultibase": "u..."}
  ],
  "authentication": ["#phone-key", "#tablet-key", "#laptop-key"],
  "capabilityInvocation": ["#phone-key"]
}
```

Note: Only the primary device holds `capabilityInvocation` rights (can modify the DID Document). Secondary devices can authenticate but not manage the identity.

### Device Enrollment Protocol

```
1. Primary device (phone) initiates pairing
2. Phone generates QR code containing:
   - DID
   - One-time pairing challenge (random, time-bound)
   - Registry endpoint URL
3. New device (tablet) scans QR code
4. Tablet generates its own keypair in local vault
5. Tablet sends to phone (via registry relay — the registry acts as a message broker using the pairing challenge as a session key; direct P2P channels like Bluetooth/Wi-Fi Direct are not supported in v1):
   - Tablet's public key
   - Signed pairing challenge (proves possession of new key)
6. Phone verifies pairing challenge signature
7. Phone builds updated DID Document (adds tablet key to authentication)
8. Phone signs DID Document update with capabilityInvocation key
9. Registry stores updated DID Document
10. Tablet now authenticates as the same DID using its own key
```

### Device Revocation

If a device is lost:
1. Primary device signs a DID Document update removing the lost device's key
2. Registry stores updated document
3. Lost device's key is immediately invalid — no token to expire, no session to invalidate

Reference: Apple (multiple devices, one Apple ID with per-device keys), FIDO2/WebAuthn (multiple authenticators per account)

## 7. Gap 4: Credential Revocation (High)

### Problem

Once a Verifiable Credential is issued, it cannot be revoked. A fired employee still holds a valid "employee" credential until it expires.

### Approach: Bitstring Status List (W3C Standard)

A single bitstring where each bit position corresponds to a credential index. Bit 0 = valid, bit 1 = revoked. The status list is published as a Verifiable Credential signed by the issuer.

### Credential with Status Reference

```json
{
  "type": ["VerifiableCredential"],
  "issuer": "did:ssdid:issuer123",
  "credentialSubject": { ... },
  "credentialStatus": {
    "id": "https://registry.example/api/status/1#42",
    "type": "BitstringStatusListEntry",
    "statusPurpose": "revocation",
    "statusListIndex": "42",
    "statusListCredential": "https://registry.example/api/status/1"
  }
}
```

### Status List Credential

```json
{
  "type": ["VerifiableCredential", "BitstringStatusListCredential"],
  "issuer": "did:ssdid:issuer123",
  "credentialSubject": {
    "type": "BitstringStatusList",
    "statusPurpose": "revocation",
    "encodedList": "uGZIP_compressed_base64url_bitstring"
  }
}
```

### Properties

| Property | Value |
|---|---|
| Privacy | Verifier downloads entire list; issuer does not learn which credential was checked (unlike OCSP) |
| Efficiency | One bitstring covers 100K+ credentials, GZIP-compressed to ~2KB |
| Offline support | Cache the status list, verify without network |
| Standard | W3C Bitstring Status List v1.0 (2024) |

### Registry Extension

Add a `/api/status/:list_id` endpoint to the registry. Issuers publish and update status list VCs through this endpoint. Verifiers fetch and cache status lists.

## 8. Gap 5: Backup & Export (High)

### Problem

Keys are trapped on the device. No way to export, migrate, or create verifiable backups.

### Approach: Encrypted Portable Backup

#### Backup Creation

> **Mobile caveat:** On Android, the AES wrapping key in Android Keystore is hardware-bound and non-exportable. The mobile backup path must: (1) authenticate the user via biometric to unlock the hardware key, (2) decrypt the private key in memory, (3) immediately re-encrypt with the passphrase-derived backup key, (4) zero the plaintext private key from memory. This requires a dedicated software export path in VaultImpl.

```
1. Authenticate user (biometric) to unlock hardware keystore
2. Decrypt raw private keys in memory (for each identity)
3. Generate random salt (32 bytes)
4. Derive backup encryption key: Argon2id(user_passphrase, salt, t=3, m=64MB, p=1)
5. Derive separate keys: enc_key = HKDF(backup_key, "enc"), mac_key = HKDF(backup_key, "mac")
6. Serialize all identities + raw private keys to JSON
7. Encrypt: AES-256-GCM(enc_key, nonce, serialized_keys)
8. Zero raw private keys from memory immediately
9. Package:
   {
     "version": 1,
     "salt": "base64url",
     "nonce": "base64url",
     "ciphertext": "base64url",
     "algorithms": ["ed25519"],
     "dids": ["did:ssdid:abc123"],
     "created_at": "2026-03-07T12:00:00Z"
   }
10. Compute HMAC-SHA256 over entire package (using mac_key — separate from enc_key)
11. Prepend HMAC to package
```

Note: HKDF key separation prevents key reuse across AES-GCM encryption and HMAC. While AES-GCM provides authenticated encryption (the tag is sufficient), the outer HMAC protects the unencrypted package metadata (salt, nonce, version) and allows integrity verification before attempting decryption.

#### Backup Recovery

```
1. User enters passphrase
2. Derive backup key from salt + passphrase (same Argon2id parameters)
3. Verify HMAC
4. Decrypt ciphertext with AES-256-GCM
5. Deserialize KeyRecords
6. Import into new vault
7. Re-activate keys with original passwords
```

#### Platform Integration

| Platform | Primary Backup Location | Alternative |
|---|---|---|
| iOS | iCloud Keychain (automatic) | Encrypted file in Files app |
| Android | Google Backup (encrypted) | Encrypted file export |
| Desktop | Encrypted file to USB/cloud | QR code for small key sets |

Reference: Signal (registration lock PIN + encrypted backup), 1Password (emergency kit), Bitcoin wallets (BIP39 seed phrase)

## 9. Gap 6: Offline Verification (Medium)

### Problem

Verifiers cannot resolve DIDs when the network is down. This blocks use cases like airport immigration, rural areas, and underground facilities.

### Approach: DID Document Bundling + Periodic Sync

#### Online Mode (Current)

```
Verifier → Registry → DID Document → verify signature
```

#### Offline Mode (New)

```
Holder presents: {
  credential,
  did_document,           ← holder bundles their current DID Document
  status_list_snapshot    ← cached status list for revocation check
}

Verifier checks:
  1. Credential signature valid against bundled DID Document?
  2. DID Document internally consistent (valid proofs)?
  3. Credential not revoked in status list snapshot?
  4. Freshness acceptable for this use case?
```

#### Freshness Policy

Each offline verification records a "last verified online" timestamp. Business rules determine acceptable staleness:

| Use Case | Max Staleness |
|---|---|
| Building access | 24 hours |
| Professional license check | 7 days |
| Emergency identification | 30 days |
| Government border control | Real-time (online required) |

#### Verifier Pre-Fetch

Verifiers that anticipate offline operation pre-fetch DID Documents and status lists for expected subjects during their last online window.

**Note**: The mobile app design document (Section 7) states "Online only" as a design decision. Offline verification is a future capability that requires: (1) a local DID Document cache, (2) bundled status list snapshots, and (3) freshness policy enforcement. The mobile app should not implement offline verification until Gap 4 (credential revocation) is complete, as offline verification without revocation checking is a security risk.

Reference: EU Digital COVID Certificate (offline verification with periodic key sync), ICAO eMRTD (passport chips carry their own certificate chain)

## 10. Gap 7: Registry High Availability (Medium)

### Problem

Single-node Mnesia deployment is a single point of failure for DID resolution.

### Approach: Phased — Hot Backup First, Clustering Later

#### Phase 1: Single Node with Automated Backup (Current Scale)

Full Mnesia clustering is unnecessary for < 1K DIDs on a single VPS.

```
Primary:  Contabo VPS Singapore — active
Backup:   Automated Mnesia backup every 6 hours to object storage
Recovery: Automated restore script (target: < 5 min RTO)
```

Backup procedure:
```elixir
# Mnesia backup to file
:mnesia.backup('/var/lib/ssdid/backups/mnesia_YYYYMMDD_HHMMSS.bak')

# Upload to object storage (S3-compatible)
# Retain last 7 daily + 4 weekly backups
```

#### Phase 2: Two-Node Mnesia Cluster (Growth)

When availability requirements increase, add a second node:

```
Node 1: ssdid_registry@vps1 (disc_copies) — Singapore
Node 2: ssdid_registry@vps2 (disc_copies) — Tokyo or KL
Mnesia: Automatic sync between nodes
DNS:    Round-robin or health-check-based failover
```

The clustering code already exists in `application.ex`. The gap is operational: documented failover procedures, split-brain resolution strategy, and monitoring.

#### Challenge Store Persistence

Current: `ram_copies` (lost on restart)
Option A: Accept the trade-off — challenges are short-lived (5 min TTL), restart invalidates pending challenges only
Option B: Move to `disc_copies` for persistence across restarts
Recommendation: Option A for simplicity. Document the behavior.

## 11. Gap 8: Rate Limiting (Medium)

### Problem

No rate limiting on any endpoint. Allows DID enumeration, challenge flooding, and registration spam.

### Approach: Token Bucket per DID + IP (ETS-Based)

```elixir
# Plug pipeline configuration
plug SsdidRegistryWeb.RateLimiter,
  rules: [
    {:challenge_create, 10, :per_minute, :per_did},
    {:did_register,      5, :per_minute, :per_ip},
    {:did_resolve,     100, :per_minute, :per_ip},
    {:did_update,        5, :per_minute, :per_did},
    {:did_deactivate,    3, :per_minute, :per_did}
  ]
```

Implementation: Use the Hammer library (Elixir) with ETS backend. No external dependencies. Counters reset automatically on window expiry.

Response for rate-limited requests:
```
HTTP 429 Too Many Requests
Retry-After: <seconds>
{"error": "Rate limit exceeded"}
```

## 12. Implementation Phases

### Phase Dependencies

```
Gap 10 (credentialStatus model) ──► Gap 4 (credential revocation) ──► Gap 6 (offline verification)
Gap 9 (nextKeyHash model)       ──► Gap 2 (key rotation)
Gap 5 (backup)                  ──► Gap 1 Tier 2 (social recovery — backup key is the SSS input)
Gap 3 (multi-device)            ──► Gap 2 (rotation needed to add/remove device keys safely)
```

### Phase 1: MVP (Before Production)

| Item | Priority | Effort | Dependencies |
|---|---|---|---|
| `VerifiableCredential` credentialStatus field (Gap 10) | Critical | Low | None |
| `DidDocument` nextKeyHash field (Gap 9) | Critical | Low | None |
| Recovery key at identity creation (Gap 1 Tier 1) | Critical | Medium | None |
| Encrypted backup/export (Gap 5) | High | Medium | None |
| Rate limiting — Hammer + ETS (Gap 8) | Medium | Low | None (backend only) |
| Activity history persistence (Gap 9) | Medium | Medium | None |
| Challenge store trade-off documentation | Medium | Low | None |

**Goal**: No identity can be permanently lost. Basic abuse prevention. Model prerequisites for Phase 2.

### Phase 2: Production Hardening

| Item | Priority | Effort | Dependencies |
|---|---|---|---|
| Key rotation with pre-commitment (Gap 2) | Critical | High | Gap 9 (nextKeyHash model) |
| Multi-device enrollment protocol (Gap 3) | High | High | Gap 2 (rotation for key management) |
| Bitstring status list for revocation (Gap 4) | High | Medium | Gap 10 (credentialStatus model) |
| Mnesia backup automation (Gap 7) | Medium | Low | None |

**Goal**: Full key lifecycle management. Credential governance.

### Phase 3: Scale & Enterprise

| Item | Priority | Effort | Dependencies |
|---|---|---|---|
| Social recovery — Shamir's (Gap 1 Tier 2) | High | High | Gap 5 (backup infrastructure) |
| Offline verification bundles (Gap 6) | Medium | Medium | Gap 4 (revocation — required for safe offline checks) |
| Multi-node Mnesia clustering (Gap 7) | Medium | Medium | None |
| Institutional recovery (Gap 1 Tier 3) | Medium | Medium | None |

**Goal**: Enterprise readiness. Offline capability. High availability.
